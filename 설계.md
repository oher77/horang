## 1. SQLite DDL 전체

### 1.1 DB 파일 구성 (2-DB 아키텍처)

- **`content.db`** (읽기 전용, 앱 에셋 번들): 단어·뜻·예문·쓰기문제. 앱 업데이트 시 통째 교체.
- **`user.db`** (읽기/쓰기, `FileSystem.documentDirectory`): Day 단어장, 학습 진행, 인출단계, 시험/점수, Income. **앱 업데이트로 절대 덮어쓰지 않음.**
- 두 DB는 `expo-sqlite`에서 각각 `openDatabaseAsync('content.db')`, `openDatabaseAsync('user.db')`로 연다.
- 교차 참조는 **애플리케이션 레벨 조인**(user.db에 `content_word_id` 정수만 저장, JS에서 두 DB 결과를 합침)으로 처리한다. `ATTACH DATABASE`는 expo-sqlite에서 동작하나, 에셋 DB 경로 변동·마이그레이션 복잡도 때문에 **기본은 앱 레벨 조인, ATTACH는 성능 병목 확인 시에만.**

> 왜 content_word_id를 "전역 안정 ID"로 쓰나: 앱 업데이트로 content.db가 바뀌어도 user.db가 참조하는 단어가 유지돼야 한다. 따라서 word의 PK는 **자동증가 rowid가 아니라, 파이프라인이 부여하는 안정적 `stable_id`**여야 한다(§3.4).

### 1.2 `content.db` DDL

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;

-- 콘텐츠 버전 (앱이 로드 시 확인, 마이그레이션 판단용)
CREATE TABLE content_meta (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- 예: ('content_version','2026.07.07'), ('word_count','2410'), ('schema_version','1')

-- 표제어-파생어 묶음(참고 메타데이터). 노출/출제 우열 없음.
CREATE TABLE lemma_group (
  id           INTEGER PRIMARY KEY,
  head_lemma   TEXT NOT NULL         -- 정규화된 표제어 문자열(표시용 그룹 라벨)
);

-- 단어 (표제어·파생어 통합, 전부 동등한 독립 항목)
CREATE TABLE word (
  id             INTEGER PRIMARY KEY,      -- = stable_id. 파이프라인이 결정, 절대 재사용 금지
  headword       TEXT NOT NULL,           -- 정규화된 영단어 (표시/TTS 대상). UNIQUE
  lemma_group_id INTEGER REFERENCES lemma_group(id),
  origin         TEXT NOT NULL,           -- 'headword' | 'derivative'
  source_row     INTEGER,                 -- 원본 xlsx 행 인덱스(추적용, nullable)
  pos_hint       TEXT,                    -- 원본 품사주석 정규화값 예: 'v' / 'n,v' (nullable)
  needs_review   INTEGER NOT NULL DEFAULT 0, -- 파이프라인 검수 플래그(0/1). 배포본은 전부 0 목표
  UNIQUE(headword)
);
CREATE INDEX idx_word_group  ON word(lemma_group_id);
CREATE INDEX idx_word_origin ON word(origin);

-- 뜻 (품사별 여러 행). 난이도 무관(뜻 자체는 난이도로 안 나눔; 예문만 난이도별).
CREATE TABLE meaning (
  id         INTEGER PRIMARY KEY,
  word_id    INTEGER NOT NULL REFERENCES word(id),
  pos        TEXT NOT NULL,               -- 'n'|'v'|'a'|'ad'|'prep'|'conj'|'phrase' 등 통제어휘
  meaning_ko TEXT NOT NULL,               -- 한국어 뜻(품사별 주요 의미)
  sort_order INTEGER NOT NULL DEFAULT 0   -- 표시 순서(주요뜻 우선)
);
CREATE INDEX idx_meaning_word ON meaning(word_id);

-- 예문 (품사별 × 난이도별). 뜻과 느슨히 연결(meaning_id nullable; 품사로 매칭도 가능)
CREATE TABLE example (
  id          INTEGER PRIMARY KEY,
  word_id     INTEGER NOT NULL REFERENCES word(id),
  meaning_id  INTEGER REFERENCES meaning(id),   -- 어느 뜻의 예문인지(가능하면 채움)
  pos         TEXT NOT NULL,                     -- meaning.pos와 일치
  level       INTEGER NOT NULL,                  -- 1=고1, 2=고2, 3=고3
  en          TEXT NOT NULL,                     -- 영어 예문(수능 스타일)
  ko          TEXT,                              -- 해석(선택)
  sort_order  INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_example_word_level ON example(word_id, level);

-- 영문쓰기 문제 (쓰기 컬럼 43행 파생). 단어와 연결.
CREATE TABLE writing_item (
  id           INTEGER PRIMARY KEY,
  word_id      INTEGER REFERENCES word(id),      -- 목표 단어(확정 시). 미확정이면 NULL + needs_review
  kind         TEXT NOT NULL,                    -- 'spelling'|'inflection'|'phrase'|'usage'
  prompt_ko    TEXT,                             -- 문제 제시문(예: '두배의', 'catch의 동사변형')
  hint         TEXT,                             -- 스펠링 힌트(예: 'd______') nullable
  answer       TEXT,                             -- 정답 문자열(스펠링='double', 변형='caught'…) 검수로 채움
  answer_alt   TEXT,                             -- 대체정답 콤마구분(다의어 변형 등) nullable
  needs_review INTEGER NOT NULL DEFAULT 0,
  raw          TEXT NOT NULL                     -- 원본 쓰기 셀 문자열(감사/재파싱용)
);
CREATE INDEX idx_writing_word ON writing_item(word_id);
CREATE INDEX idx_writing_kind ON writing_item(kind);
```

### 1.4 날짜 저장 포맷 — 규칙 (중요)

- **`*_day` (epoch day)**: `Math.floor(epochMs_of_local_midnight / 86400000)` 로 저장. 실제로는 **로컬 자정을 기준으로 한 정수 일련번호**. 복습 "-N일 전 학습한 Day"는 `WHERE created_day = :today_day - :offset` 로 **인덱스 탄 정수 등가 비교**. 오프셋 7개는 `IN (:d1,...,:d7)`.
- **`*_ms` (epoch ms)**: `Date.now()` 스냅샷. 로컬 표시·"최근 5일"·월별 그룹핑용. 월별 집계는 JS에서 ms→`YYYY-MM` 파생하거나, SQLite `strftime('%Y-%m', taken_ms/1000, 'unixepoch', 'localtime')` 로 그룹핑(인덱스는 `taken_ms` 범위로 좁힌 뒤 그룹).
- **"최근 5일 점수"**: `WHERE taken_day >= :today_day - 4 ORDER BY taken_ms DESC` → `idx_session_takenday` 사용.
- **왜 문자열(ISO)이 아닌가**: `-N일` 계산이 정수 뺄셈 한 번으로 끝나고(문자열은 날짜연산 불가), 타임존/DST로 인한 "하루 경계" 버그를 로컬자정 고정으로 차단. 저장 두 컬럼은 중복이지만 각각 다른 쿼리축(정수 등가 vs 범위/포맷)을 인덱스로 최적화하기 위한 의도적 비정규화.
- **주의**: `_day`는 반드시 **디바이스 로컬 자정** 기준으로 계산. 헬퍼 `toEpochDay(date=new Date())`를 단일 유틸로 두고 전 코드가 그것만 쓴다(타임존 일관성).

## 2. 원본 정제 + 쓰기 컬럼 파싱 규칙

원본: `data/voca_data.xlsx` Sheet1, 2212행, 컬럼 `[Unnamed:0(번호), 표제어, 파생어, 쓰기]`.
탐색 재확인 결과(이번 세션): 완전빈행 8곳 = 행 인덱스 **[921,1004,1014,1210,1559,1616,1619,1627]**. 표제어 non-null 2169 / 파생어 non-null 205(분해 시 ~241) / 쓰기 non-null 43(표제어와 동시 14, 표제어 없이 29, 파생어와 동시 1).

### 2.1 전역 정규화 (모든 문자열 공통)

1. 유니코드 정규화: `NFKC`. 소프트하이픈(U+00AD, `half\xadmoon` 1건) 및 제어/포맷 문자(category `Cf`) 제거. `half\xadmoon` → `halfmoon`. (NFKC가 다수 호환문자를 정규화하나 U+00AD는 별도 제거 필요 → 명시적으로 `strip(chr(0xAD))`.)
2. 공백 정리: 양끝 trim, 내부 연속공백 1칸.
3. 스마트 인용부호·전각→반각(NFKC가 대부분 처리).

### 2.2 행 필터

- **완전 빈 행 8곳 삭제** (위 인덱스).
- **번호 컬럼 무시**.

### 2.3 표제어 정규화 (weird_head 51건 규칙화)

원본 표제어 51건이 순수 알파벳이 아님. 유형별 처리:

| 유형 | 예 | 처리 |
|---|---|---|
| 품사주석 `word (v)` `(a,n)` `(n, v)` | `board (v)`, `fat (a,n)`, `land(n, v)` | 괄호 앞 단어를 `headword`로, 괄호 안을 `pos_hint`로 분리(정규화: 소문자, 콤마구분, `형용사`류 한글 주석은 무시/드롭). `land(n, v)`처럼 붙은 케이스도 정규식 `^([A-Za-z]+)\s*\(([^)]*)\)` 로 처리 |
| 철자 변이 `A/ B` | `advertize/ advertise`, `analyze/ analyse`, `defense/ defence` | 슬래시 분리 → **각각 독립 word 행**(둘 다 유효 스펠링). 같은 `lemma_group` 부여. 선두 토큰을 그룹 라벨 |
| 다의어 주석 | `lie (다의어)`, `liberal (다의)`, `fashion (다의 v, n)` | 괄호 안 한글 주석은 드롭, 앞 단어만 headword. "다의"는 뜻 생성 단계에서 품사 다수 채우라는 신호로 `pos_hint`에 보존 가능 |
| 구/숙어(유효 학습 대상) | `in front of`, `next to`, `keep ~ing`, `so ~ that`, `too ~ to`, `deprive A of B` | **headword로 유지**(구 단위 표제어). `origin='headword'`, `pos_hint='phrase'`. TTS는 구 전체 읽기 |
| 한국어 뜻만(표제어 오염) | `커플, 한쌍`(407), `도너츠`(582), `과일`(848) | 목표 영단어 미기재. **수동보정 큐로 이관**(`needs_review=1`, headword는 임시로 원문 유지 or 추정 영단어). 알파벳 위치로 추정: 407은 couple 근처, 582 donut, 848 fruit — 검수자가 확정 |
| 지시문/메모(단어 아님) | `문장에서 often의 위치는?`(별도), `such가 형용사를 강조하는 예문 3개`(2033), `ingredient 발음`(1058), `party (파티 말고 뜻)` | **word로 만들지 않음.** 이런 건 "이미 존재하는 다른 단어에 대한 부가 지시"거나 노이즈. 처리: `often`·`ingredient`·`party` 등 대상 단어가 시트에 이미 있으므로 **해당 단어의 보강 노트/writing_item(usage)로 흡수**하거나 드롭. 수동보정 큐에 근거와 함께 남김 |
| 문장/감탄 | `none of your business!`(187), `marry me`(1252), `passed away`(1483), `I'm moved`(1343), `on your feet`(814), `on sale...` | 표현 학습 대상으로 볼지 판단 필요 → **수동보정 큐**. 기본 정책: 대표 핵심어(`business`,`marry`,`pass`…)가 이미 있으면 예문/구 표현으로 흡수, 아니면 phrase headword로 채택 |

정규식 핵심:
```
POS_ANNOT   = r'^([A-Za-z][A-Za-z\-\s]*?)\s*\(([^)]*)\)\s*$'   # 'board (v)', 'fat (a,n)'
SLASH_VAR   = r'^\s*([A-Za-z]+)\s*/\s*([A-Za-z]+)\s*$'          # 'defense/ defence'
PLAIN_WORD  = r'^[A-Za-z][A-Za-z\-]*$'
PHRASE_HINT = 표제어에 공백 다수 + 물결(~)/대문자 A,B 슬롯 → phrase
```
헝글 포함 → 한국어 뜻/지시문 → 수동보정 큐 후보.

### 2.4 파생어 정규화 (205건 → ~241 독립 항목)

- 대부분 `(abortion)` 괄호형(192건). 괄호 제거 → 단어 추출.
- 비괄호 13건 처리:
  - 콤마/슬래시 다중: `maths / math`, `interviewee, interviewer`, `On sale, For sale, Clearance sale`, `in terms of, terms and conditions` → **분리해 각각 독립 word**.
  - 선두 슬래시 `/ practise`, `/ theatre`, `/ towards`, `/ upwards`, `/ utilise` → 선두 `/` 는 "표제어의 영국식 철자"를 뜻함. 선두 `/` 제거 후 단어화, 같은 행 표제어와 같은 `lemma_group`.
  - `half\xadmoon` → NFKC+softhyphen 제거 → `halfmoon`.
  - 단독 단어 `sentimental`, `thirsty`, `plastic surgery` → 그대로 word(구는 phrase).
- 분해 규칙: `strip('()')` → `chr(0xAD)` 제거 → `re.split(r'[,/]', s)` → 각 토큰 trim·선두`/`제거 → 빈토큰 버림 → `PLAIN_WORD`면 word, 공백 포함이면 phrase.
- 각 파생 word: `origin='derivative'`, 같은 원본행 표제어가 있으면 그 표제어의 `lemma_group_id` 공유(없으면 자기 자신이 그룹 헤드).
- **중복 병합**: 표제어와 파생 분해 결과가 겹치면(예: 이미 word로 존재) `UNIQUE(headword)`로 1개만. 병합 시 `origin`은 'headword' 우선.

### 2.5 쓰기 컬럼 파싱 (43건) — 형식 a~e 규칙

목표: 각 쓰기 셀 → `writing_item` 행. **목표 단어(word_id) 확정**이 핵심.

먼저 **같은 행 표제어 유무**로 1차 분기:
- 표제어 있음(14건): `word_id` = 그 표제어의 word. 예 `catch|catch의 동사변형`, `hide|hide의 동사변형`, `lie (다의어)|lie 의미별 동사변형`. → 자동 확정 가능.
- 표제어 없음(29건): 목표 단어 미기재 → 파싱+추론 후에도 사람 검수.

형식별 규칙:

| 형식 | 예 | kind | 목표단어 결정 | answer |
|---|---|---|---|---|
| **a) 뜻+스펠링힌트** | `두배의: d______`, `축제: f_______`, `영화 m_____` | `spelling` | 힌트 첫글자+알파벳위치로 강력추정(d→double, f→festival, m→movie). **검수 승인 필수** | 검수자가 확정 스펠링 입력 |
| **b) 문법변형 메모** | `eat의 과거형`, `catch의 동사변형`, `man의 복수형`, `fly의 진행형`, `leaf의 복수형` | `inflection` | **메모 안의 영단어가 곧 원형**(정규식 `^([A-Za-z]+)\s*(의|\s)`로 추출). eat/catch/man/fly/leaf. 표제어 없어도 자동 확정 가능 | 변형형 정답을 LLM/검수로 채움(ate, caught, men, flying, leaves). `lie 의미별`은 다의 → `answer_alt` |
| **c) 한국어 뜻만** | `가위`, `무거운`, `털, 머리카락`, `햄버거`, `병원`, `주스`, `기계`, `운`, `비밀, 비밀의` | `spelling` | 뜻+알파벳위치로 추정(가위=scissors[scholar~scope], 무거운=heavy[heaven~heel], 털=hair[habitat~half]…). **자동확정 불가 → 수동보정 큐 전량** | 검수자 확정 |
| **d) 구/관용 빈칸** | `(   ) a break: 휴식하다`(break), `(   ) care: 잘 지내세요`(care), `예를 들어: (   ) example`(example), `(   ) a break`류 | `phrase` | 같은 행 표제어 있음 → 그 단어. 빈칸 답은 `take`(a break)/`take`(care)/`for`(example) 등 **문맥어** → LLM/검수 | 빈칸 정답 문자열 |
| **e) 서술형/용법 메모** | `문장에서 often의 위치는?`, `one 의 쓰임에 대해 (숫자말고)`, `일주일에 3시간`, `매주 수요일` | `usage` | 영문쓰기 채점 부적합(정답 단일 아님). 대상어(often, one) 있으면 그 word에 `usage` 노트로. 없으면(`일주일에 3시간`,`매주 수요일`) **드롭 or 수동보정** | 보통 NULL, 참고 프롬프트만 |

파싱 파이프라인 산출:
1. 자동확정(word_id 채움, needs_review=0): 형식 b 전부(원형이 메모에 명시) + 표제어 있는 a/c/d.
2. 추정(word_id 채우되 needs_review=1): 표제어 없는 a(스펠링힌트).
3. 수동보정 큐(word_id NULL, needs_review=1): 표제어 없는 c(한글뜻만), e 중 대상어 불명(`일주일에 3시간`,`매주 수요일`).
4. `raw`에 원본 셀 전량 보존(재파싱 가능).

**수동보정 큐 산출물**: 파이프라인이 `review/writing_manual.csv`(컬럼: `raw, guessed_word, prev_head, next_head, kind, note`)로 뽑는다. 알파벳 위치(prev/next 표제어)를 힌트로 함께 실어 검수자가 목표단어를 확정한다. 이번 세션에서 확인한 29건의 prev/next 이웃 표제어는 파이프라인이 자동 계산(§본문 탐색 결과와 동일 로직: 위/아래 최근접 non-null 표제어).

### 2.6 정규화 후 예상 규모

- word: 표제어 유효분(~2160, 노이즈/구 제외분 감안) + 파생 분해(~241) − 중복병합 ≈ **2300~2400행**.
- meaning: 단어당 1~3 품사 → ~5000행.
- example: word × pos × 3난이도 → **수천~1.5만행** (수 MB, 로컬 저장 무리 없음 — 기술방향 §7 확정과 일치).
- writing_item: ~43행(일부 드롭 후 ~35).

### 3.1 전체 흐름 (오프라인, 개발자 머신에서 1회 실행 → 산출물 커밋)

```
data/voca_data.xlsx
   │  (1) clean.py  — §2 정제·파싱
   ▼
build/words.json            (정제된 word/lemma_group/writing_item, stable_id 부여)
review/writing_manual.csv   (수동보정 큐)
review/word_review.csv      (needs_review=1 word 목록: 한글뜻·문장 등)
   │  (2) 사람 검수 → review/*.csv 수정본 반영 → clean.py 재실행(idempotent)
   ▼
build/words.final.json
   │  (3) gen_content.py — LLM 일괄 호출. word별 (뜻 품사별 + 예문 3난이도) 생성
   ▼
build/content.json          (검수용 중간 산출물: word→meanings→examples)
review/content_sample.csv   (샘플 N개 사람 검수)
   │  (4) 사람 스팟체크(난이도 적정성·오역·수능스타일)
   ▼
build/content.json (승인)
   │  (5) pack_db.py — content.json → content.db (SQLite 파일 생성, §1.2 DDL)
   ▼
assets/db/content.db  ← 앱 번들에 포함(git-lfs 권장)
```

- **(1)(2)(5)는 결정론적·재실행 안전(idempotent)**. LLM 호출 (3)만 비용 발생 → 1회 대량 배치.
- LLM 생성은 **Claude API 배치**(§claude-api 스킬 참조로 모델/배치 API 확정). 스택은 앱 런타임에 LLM을 부르지 않음(오프라인 내장) — 기술방향 §7·확정사항 준수.

### 3.2 LLM 생성 프롬프트 규격 (gen_content.py)

- 입력: `word.headword`, `pos_hint`, (있으면)기존 뜻 힌트.
- 출력(JSON 강제): `{ meanings:[{pos, meaning_ko, sort_order}], examples:[{pos, level(1|2|3), en, ko, sort_order}] }`.
- 제약: 예문은 수능 스타일, `level`별 어휘·문장길이 차등(고1 짧고 평이 → 고3 복문·추상). 품사별로 최소 1예문×3난이도. 부적절어 필터.
- 검수 중간산출물: `content.json`은 사람이 CSV로 열어 스팟체크. 전량 검수는 비현실적이므로 **품사 커버리지·난이도 단조성(고1≤고2≤고3 길이) 자동 검증 + 무작위 표본 사람 검수**.

### 3.3 앱 탑재 방식 — 트레이드오프 & 결정

| 축 | (A) 프리빌드 .db 에셋 번들 | (B) 첫 실행 JSON→DB 구축 |
|---|---|---|
| 콜드스타트 | 즉시(파일 복사 1회) | 수천~1.5만 INSERT → 수 초 지연 + 진행바 필요 |
| 앱 번들 크기 | .db(수 MB) 포함 | .json(유사 크기, gzip 유리) 포함 |
| 무결성/인덱스 | 빌드시 확정, 런타임 리스크 0 | 런타임 스키마/인덱스 생성 실패 여지 |
| 업데이트 교체 | 파일 교체 단순 | 재구축 재실행 |
| Expo 자산 접근 | `Asset.fromModule` → `FileSystem.copyAsync`로 documentDir 복사 후 open | 번들 JSON require |

**결정: (A) 프리빌드 .db 에셋 번들.** 이유: (1) 콜드스타트가 곧 UX(스플래시 뒤 바로 첫 단어장). (2) content.db는 읽기 전용이라 documentDir로 1회 복사만 하면 됨. (3) 무결성이 빌드타임에 고정돼 런타임 실패면이 사라짐. (B)의 유일한 장점(번들 크기 gzip)은 수 MB 수준이라 무의미.

탑재 절차(런타임):
```
1. 앱 시작 시 documentDir/content.db 존재 & content_version 확인
2. 없거나 번들 버전 > 저장 버전 → Asset(assets/db/content.db) → copyAsync(documentDir/content.db) 덮어쓰기
   (content.db는 읽기전용이므로 사용자 데이터 손실 없음)
3. openDatabaseAsync('content.db'), 별도로 openDatabaseAsync('user.db')
```

### 3.4 앱 업데이트 시 진행 데이터 보존 전략 (핵심)

문제: 새 단어 데이터(content.db v2)로 교체 시 user.db가 참조하는 `content_word_id`가 깨질 위험.

전략:
1. **stable_id 불변 계약**: word.id는 파이프라인이 부여하고 **버전 간 재사용/재배정 금지**. `clean.py`는 `build/id_map.json`(headword→stable_id)을 영속 저장하고, 신규 단어만 새 id(기존 max+1↑) 부여. 삭제된 단어의 id는 **결번 처리(재사용 금지)**. → user.db의 `content_word_id`는 버전 업 후에도 같은 단어를 가리킴.
2. **user.db는 앱 업데이트로 절대 덮어쓰지 않음**(문서 디렉터리, 번들과 분리).
3. **끊어진 참조 방어**: 앱 로드시 `day_word.content_word_id`가 새 content.db에 없으면(단어 삭제됨) 해당 행을 "tombstone"로 표시(별도 컬럼 or 조회 시 LEFT JOIN 결과 NULL이면 '삭제된 단어'로 렌더링). Day 재생성은 하지 않음(과거 학습 이력 보존 원칙).
4. **user_schema 마이그레이션**: `app_meta.user_schema_version`로 관리. DDL 변경 시 `migrations/NNN.sql` 순차 적용.
5. **content 마이그레이션 훅**: 뜻/예문만 개선된 마이너 업데이트는 그냥 content.db 교체로 끝(참조 id 유지). 단어 추가/삭제가 있는 메이저는 id_map으로 안전.

이유: Expo 공식 기본·파일기반이라 정적 화면 트리에 명료, 타입드 라우트/딥링크 기본 제공, 설정 보일러플레이트 적음. react-navigation은 expo-router의 하부이므로 필요시 옵션 접근 가능(손해 없음). Drawer(호랑이 메뉴)는 expo-router의 Drawer 레이아웃(`@react-navigation/drawer` 경유, Expo Go 호환) 사용.

### 4.3 상태관리 — Zustand + 화면 로컬 (최소주의)

- 전역(Zustand) 스토어는 **딱 3개만**: `dbStore`(content/user DB 핸들, ready 플래그), `settingsStore`(level, words_per_day, income_rules — 앱 시작 시 로드·설정 화면에서 갱신), `todayStore`(오늘 Day id/day_index 캐시).
- 화면별 목록 데이터(단어장 행, 리스트, 그래프 포인트)는 **화면 로컬 state + 직접 쿼리**(`useEffect`에서 로드, 뮤테이션 후 재조회). React Query 등 서버상태 캐시 라이브러리 미도입 — 소스가 로컬 SQLite뿐이라 stale/refetch/캐시 무효화 계층이 불필요(오버엔지니어링).
- 쿼리는 `db/queries.ts`에 순수 함수로 모아 재사용(§5). 스크린은 그 함수만 호출.

### 4.5 100+행 테이블 가상화 + 가림 애니메이션 전략

기획 요구: 컬럼 헤더 눈아이콘 → 컬럼 전체 가림, 셀 클릭 → 유령이 왼쪽으로 지나가며 글자가 먼지처럼 사라짐. 100+ 셀 성능 대안: "보이는 15~20행만 렌더/애니메이션 + stagger 10~20ms".

- **가상화**: RN 내장 `FlatList`(FlashList는 Expo Go 미포함 → 금지). `initialNumToRender=18`, `windowSize=5`, `maxToRenderPerBatch=8`, `removeClippedSubviews={true}`, `getItemLayout`(행 고정높이라 계산 가능 → 스크롤 점프 방지·오프스크린 언마운트 정확).
- **가림 상태 모델**: 컬럼 가림은 3개 boolean(word/meaning/example)로 헤더 아이콘이 토글. 셀 개별 가림은 각 행의 `{word:bool, meaning:bool, example:bool}` 로컬맵. 이 state는 화면 로컬(DB 미저장 — 세션성 UI 상태). 단어장 화면의 학습/인출모드 토글(§7.3)은 모드 전환 시점에 `columnHidden.meaning`을 1회성으로 세팅할 뿐 — 이후 헤더 눈아이콘 조작은 이 state를 독립적으로 계속 토글한다.
- **애니메이션(reanimated)**: 각 셀 위에 "먼지/유령" 오버레이 레이어를 얹고(텍스트 자체 파티클화 금지 — 기획 성능주석), 트리거 시 오버레이 `translateX`(우→좌 스윕) + 텍스트 `opacity 1→0`를 `withTiming`. **stagger**: 컬럼 일괄 가림 시 현재 화면에 보이는 행들에 대해서만 `index * 15ms` 지연(`withDelay`). 화면 밖 행은 FlatList가 언마운트했으므로 자연히 애니메이션 대상 아님(요구 충족).
- **핵심**: 애니메이션 대상 = **onViewableItemsChanged로 파악한 가시 행**뿐. 스크롤로 새로 들어온 행은 이미 목표 가림상태를 즉시 반영(애니메이션 없이 정적)하여 스크롤 성능 유지.
- 셀 클릭 개별 가림도 동일 오버레이 재사용, stagger 없이 단발.

## 5. 핵심 쿼리 예시

날짜 유틸: `const D = (offset=0) => todayEpochDay - offset;` (todayEpochDay = 로컬자정 기준 정수). content↔user는 앱 레벨 조인(2쿼리 후 JS merge) 기본.

### Q — 오늘 단어장 생성 (전 Day 중복 금지 보장)

핵심: `day_word.content_word_id UNIQUE` 제약이 **전 Day 걸친 중복을 DB가 물리적으로 보장**. 추출은 "아직 어떤 Day에도 안 쓰인 단어 중 랜덤 N개".

```sql
-- (트랜잭션 내, user.db)
-- 1) 오늘 Day 이미 있으면 그걸 사용(하루 1개 규칙)
SELECT id, day_index FROM day WHERE created_day = :today LIMIT 1;

-- 없으면 생성:
-- 2) 아직 미사용 content_word_id 후보를 content.db에서 랜덤 N개.
--    "사용됨" 집합은 user.db.day_word에 있음 → 두 DB라 앱레벨로 처리:
--    (a) user.db: SELECT content_word_id FROM day_word;  → usedSet(JS Set)
--    (b) content.db: 전체 word.id 중 usedSet 제외하고 랜덤 N.
--        대량이면 content.db에서 랜덤 넉넉히 뽑고 JS에서 usedSet 필터 후 N개 컷.
SELECT id FROM word ORDER BY RANDOM() LIMIT :N_plus_buffer;   -- content.db
--    JS: candidates.filter(id=>!usedSet.has(id)).slice(0,N)

-- 3) day insert + day_word 벌크 insert (position 0..N-1)
INSERT INTO day (day_index, created_day, created_ms, is_started, words_count)
VALUES ((SELECT COALESCE(MAX(day_index),0)+1 FROM day), :today, :nowMs, 0, :N);
-- day_word: INSERT ... (UNIQUE 위반 시 그 단어만 스킵하고 버퍼에서 보충)
```
- **경쟁 조건 없음**(단일 기기·단일 프로세스). 대량 pool에서 `ORDER BY RANDOM()`은 2400행 규모라 성능 문제 없음. 남은 미사용 단어가 N 미만이면 가능한 만큼만 생성(소진 처리).
- **words_per_day 변경 시 재생성**: `is_started=0`인 Day만 삭제(`ON DELETE CASCADE`로 day_word 정리 → 그 단어들 다시 미사용 풀로 복귀) 후 새 N으로 재구성. `is_started=1`은 불변.

### Q-REVIEW-DAYS — 복습 대상 7 오프셋

```sql
-- user.db. 오늘 기준 -1/-3/-7/-14/-30/-60/-120일에 생성된 Day, 최근순
SELECT id, day_index, created_day, created_ms
FROM day
WHERE created_day IN (:d1,:d3,:d7,:d14,:d30,:d60,:d120)   -- = today-1, today-3, ...
ORDER BY created_day DESC;
-- idx_day_created 사용. (해당 오프셋에 Day 없으면 그 항목은 결과에서 빠짐)
```

### Q-SCARY-TOP10 — 낯가림 단어 (오답 1회↑, 횟수순 상위 10)

```sql
-- user.db: 오답 횟수 집계
SELECT content_word_id, COUNT(*) AS wrong_cnt
FROM test_item
WHERE is_wrong = 1
GROUP BY content_word_id
HAVING wrong_cnt >= 1
ORDER BY wrong_cnt DESC
LIMIT 10;
-- idx_testitem_wrong(부분인덱스) 활용. 결과 word_id로 content.db에서 headword 조회(JS merge).
```

### Q-TEST-POOL — 테스트 출제 풀 (당일 + 복습대상, 랜덤)

```sql
-- user.db: 당일 Day + 복습 Day들의 day_word를 합쳐 셔플
SELECT dw.content_word_id
FROM day_word dw
JOIN day d ON d.id = dw.day_id
WHERE d.id = :todayDayId
   OR d.created_day IN (:d1,:d3,:d7,:d14,:d30,:d60,:d120)
ORDER BY RANDOM();
-- 문제/답 렌더는 content.db에서 해당 word들의 headword/뜻/writing_item 조회.
```

### Q-INCOME-MONTH — 월별 Income 합 (이달)

```sql
-- user.db. taken_ms 범위로 인덱스 좁힌 뒤 로컬월 그룹(또는 이달 경계 ms를 JS로 계산해 BETWEEN)
SELECT COALESCE(SUM(income_amount),0) AS month_income
FROM test_session
WHERE taken_ms >= :monthStartMs AND taken_ms < :nextMonthStartMs;
-- 월별 추이(Q-INCOME-TREND)는 test_session과 habit_bonus를 각각 월별로 GROUP BY한 뒤
-- JS에서 합산한다(습관 보너스도 Income에 포함되므로) — 구현: lib/statsQueries.ts의
-- getMonthlyIncomeTotals(monthCount = 6), 최근 monthCount개월을 데이터 없는 달은 0으로
-- 채워 오래된→최신 순으로 반환.
SELECT strftime('%Y-%m', taken_ms/1000,'unixepoch','localtime') AS ym,
       SUM(income_amount) AS income
FROM test_session
GROUP BY ym ORDER BY ym;
```

### Q-RECENT5 — 최근 5일 점수

```sql
-- user.db
SELECT ts.taken_ms, ts.day_id, d.day_index, ts.score100, ts.income_amount, ts.paid
FROM test_session ts JOIN day d ON d.id = ts.day_id
WHERE ts.taken_day >= :today - 4
ORDER BY ts.taken_ms DESC;
-- idx_session_takenday 사용.
```

### Q-WORD-STATE-TREND — 단어 정답/오답 상태 추이 (일별, 최근 30일, 2026-07-09 Q-CORRECT-CUMULATIVE 대체)

```sql
-- user.db: 전체 test_item ⋈ test_session을 taken_ms 오름차순 1회 읽기
SELECT ti.content_word_id, ti.is_wrong, ts.taken_day
FROM test_item ti JOIN test_session ts ON ts.id = ti.session_id
ORDER BY ts.taken_ms ASC, ti.id ASC;
```
- 의미: "지금까지 테스트된 고유 단어 중 최신 결과가 정답/오답인 단어 수"(재채점 시
  정답↔오답이 뒤집힐 수 있어 세션 단위 누적 합으로는 표현 불가 — Q-CORRECT-CUMULATIVE는
  이 도메인 규칙과 맞지 않아 폐기).
- 계산은 SQL 윈도우 함수가 아니라 JS 단일 패스로 한다: Map<content_word_id, isWrong>을
  유지하며 taken_day 경계마다 (correctCount, wrongCount) 스냅샷을 남기고, 최근 30일
  시리즈로 변환할 때 데이터 없는 날은 직전 값 이월, 데이터 시작 전은 0. 구현:
  `lib/statsQueries.ts`의 `getWordStateTrend(days = 30)`.

### 그 외

- **Q-INCOME-FOR-SCORE**: `SELECT amount FROM income_rule WHERE min_score <= :score ORDER BY min_score DESC LIMIT 1;`
- **Q-SEARCH**(단어 검색): content.db `SELECT id, headword FROM word WHERE headword LIKE :q||'%' ORDER BY headword LIMIT 50;` (뜻 검색까지면 meaning JOIN).
- **recall_stage 스와이프**: `UPDATE day_word SET recall_stage = MAX(0,MIN(5, recall_stage + :delta)) WHERE id = :id;`

## 7. 하루 4회 분산 인출 습관 시스템

목표: 학생이 하루 안에 인출(retrieval)을 여러 번 분산하도록 유도한다. 몰아치기 한 번보다
간격을 둔 4회의 짧은 인출이 장기 기억에 유리하다는 전제(분산 학습·인출 연습). 각 시간창에서
1회만 인정해 "한 번에 몰아서 4칸 채우기"를 구조적으로 차단한다.

### 7.1 개념 · 판정 규칙

**슬롯(시간창) 정의**
- 하루를 4개 시간창(슬롯)으로 나눈다. 기본값(로컬 시간): S0=06–10, S1=10–15, S2=15–20, S3=20–24.
- 슬롯 경계는 정수 시(hour). 각 슬롯은 `[start, end)` 반열림 구간(start 포함, end 제외).
- 06시 이전(00:00–05:59)은 어떤 슬롯에도 속하지 않는 **데드존**이다(4슬롯이 06–24만 커버).
  이 시간대의 인출은 세션으로 인정하지 않는다(경계 판정 단순화 + 심야 사용 억제). 새벽형 학생은
  설정에서 S0 start를 낮추면 된다(§7.3).

**"인출 1회 완료" 판정 기준 — 체류시간 단독(커버리지 폐지) — 사용자 확정 2026-07-09**
- 세션은 오늘 단어장 화면(`/day/[dayId]`)의 인출 활동으로 판정하며, 판정은 **모드(학습/인출) 무관**하고
  체류시간만 본다:
  1. **오늘 첫 세션**: 화면 진입 후 누적 활성 시간(백그라운드 이탈 시간 제외) ≥ **단어수 × 5초** (2026-07-10 사용자 조정, 최초 1초→5초).
  2. **이후 세션(둘째~넷째)**: 누적 활성 시간 ≥ **3초 + (스와이프 배지 단어 수 — recall_stage > 0) × 1초**.
     배지 수는 화면 로드 시점에 **1회만** 계산한다 — 세션 도중 스와이프로 배지 수가 바뀌어도 그 세션의
     임계값은 고정.
- 과거 존재했던 "가림을 걷어 확인한 고유 단어 수 ≥ ceil(N*0.7)" 커버리지 조건은 **폐지**됐다("숫자 늘리기"가
  단어 자체보다 목표처럼 느껴진다는 사용자 판단, 2026-07-09). 화면에도 "이번 인출 X/Y" 진행 인디케이터는
  더 이상 표시하지 않는다 — 대신 학습/인출모드 토글과 인출모드 전용 카운트다운 라인바를 둔다(§7.3).
- 조건이 처음 만족되는 순간 **자동으로** 세션을 확정한다. 별도 [완료] 버튼은 없다(매번 버튼 누르는
  마찰 제거). 조건 충족 시 토스트("이번 슬롯 인출 완료 ●")로 피드백.
- **슬롯당 1회만 인정**: 확정 시 현재 로컬시각이 속한 슬롯을 계산하고, `(local_day, slot_index)`가
  아직 로그에 없을 때만 기록한다. 이미 있으면 무시(추가 인출은 허용하되 게이지는 안 늘어남).
- **꼼수 대비 요약**: 체류 단독 조건이라 "열고 방치"를 완전히 막지 못한다 — 알려진 한계로 §7.6 미결에
  기록(최소 상호작용 하한 추가 여부는 실사용 후 판단).

**날짜 경계 — "오늘"의 정의**
- §1.4의 `*_day`는 디바이스 **로컬 자정 기준** epoch day이므로, 슬롯 판정이 요구하는 "학생의 로컬
  하루"와 정확히 일치한다. 슬롯 로그의 날짜 키도 기존 `todayEpochDay()`를 그대로 재사용하고,
  새 시간 규약은 도입하지 않는다.
- 슬롯 인덱스는 **인출을 확정하는 그 순간의 로컬 벽시계 시(hour)** 로 계산한다. 23:59 확정은 오늘 S3,
  00:10 확정은 데드존(무효), 06:10 확정은 다음 날 S0으로 자연히 귀속된다 — "오늘"의 경계는 로컬
  자정 하나로 통일(§1.4 원칙과 동일).
- 저장은 `local_day`(epoch day, 등가비교용) + `done_ms`(epoch ms, 감사·정렬용) 두 축(§1.4의 의도적
  비정규화 관행 계승). 슬롯 판정 자체는 확정 시각의 로컬 시(hour)만 쓴다.

### 7.2 user.db 스키마 추가분 + 마이그레이션

**추가 DDL**

```sql
-- 하루 4회 분산 인출 로그. 하나의 확정된 인출 세션 = 1행.
-- UNIQUE(local_day, slot_index)가 "슬롯당 1회"를 DB 차원에서 물리 보장(§7.1).
CREATE TABLE IF NOT EXISTS retrieval_session (
  id          INTEGER PRIMARY KEY,
  local_day   INTEGER NOT NULL,              -- epoch day(로컬 자정 기준, §1.4). 스트릭·게이지 조회 키
  slot_index  INTEGER NOT NULL,              -- 0..3 (설정된 슬롯 순서). 데드존은 기록 안 함
  source      TEXT NOT NULL DEFAULT 'today', -- 'today'(오늘 단어장) | 'review'(복습) — 확장 여지(§7.6)
  day_id      INTEGER REFERENCES day(id) ON DELETE SET NULL, -- 어느 Day를 훑었는지(감사용)
  done_ms     INTEGER NOT NULL,              -- 확정 시각 epoch ms
  UNIQUE(local_day, slot_index)              -- ★ 슬롯당 1회 인정(핵심 제약)
);
CREATE INDEX IF NOT EXISTS idx_retrieval_day ON retrieval_session(local_day);

-- 슬롯 경계 설정(4행 고정, slot_index 0..3). settings 단일행에 컬럼 8개로 붓는 대신 행 테이블 —
-- 관리가 깔끔하고 향후 슬롯 수 조정 여지를 남긴다. 겹침 금지 등 쌍 간 검증은 UI 층(§7.3).
CREATE TABLE IF NOT EXISTS slot_config (
  slot_index  INTEGER PRIMARY KEY CHECK (slot_index BETWEEN 0 AND 3),
  start_hour  INTEGER NOT NULL CHECK (start_hour BETWEEN 0 AND 23),
  end_hour    INTEGER NOT NULL CHECK (end_hour BETWEEN 1 AND 24),  -- 24 = 자정
  CHECK (start_hour < end_hour)
);
-- 기본값 시드: (0,6,10)(1,10,15)(2,15,20)(3,20,24)

-- 습관 보너스 지급 스냅샷(§7.4). 시험 점수와 무관하므로 test_session에 섞지 않는다.
CREATE TABLE IF NOT EXISTS habit_bonus (
  id         INTEGER PRIMARY KEY,
  local_day  INTEGER NOT NULL,
  kind       TEXT NOT NULL,                  -- 'full_day' | 'streak7'
  amount     INTEGER NOT NULL,               -- 지급 시점 스냅샷(상수가 바뀌어도 소급 안 함)
  paid       INTEGER NOT NULL DEFAULT 0,     -- 부모 지급 체크(test_session.paid와 동일 개념)
  created_ms INTEGER NOT NULL,
  UNIQUE(local_day, kind)                    -- 하루 같은 종류 보너스 중복 지급 방지(멱등)
);
```

**마이그레이션 절차(중요 — user.db는 이미 실기기에 존재)**

현재 `ensureUserDb()`(lib/db.ts)는 전 테이블을 `CREATE TABLE IF NOT EXISTS`로 적용하고 스키마 버전을
`app_meta('user_schema_version')`에 `INSERT OR IGNORE`로 기록한다(`PRAGMA user_version` 미사용).
마이그레이션도 이 관행을 그대로 잇는다:

1. `USER_DB_DDL`에 위 CREATE TABLE(IF NOT EXISTS) 3개 + 인덱스를 추가한다. 멱등이므로 신규·기존
   설치 모두 안전(기존 테이블 불변).
2. `slot_config` 기본 4행 시드는 `ensureIncomeRules()`와 동일한 **lazy seed** 패턴: `COUNT(*)=0`일
   때만 4행 INSERT(트랜잭션).
3. 스키마 버전 `'1'→'2'`: `INSERT OR IGNORE`는 기존 행을 못 올리므로 DDL 적용 뒤
   `UPDATE app_meta SET value='2' WHERE key='user_schema_version'`를 실행한다(신규 설치 seed도 '2'로).
4. 신규 테이블은 기존 데이터와 무관 — 백필 불필요. 과거 학습 이력에서 슬롯 로그를 소급 생성하지
   않는다(당일 시각 데이터가 없어 슬롯 귀속 불가 — 기능 도입일부터 스트릭 시작).
- **실패 안전성**: 모든 추가가 IF NOT EXISTS + lazy seed라 재실행 멱등. 중간 실패 시 다음 앱 시작에서
  이어서 완성된다.
- **'2'→'3' (2026-07-09)**: `pron_resolution`(발음 헷갈림 해소 기록, §6-10) 추가 — 동일 관행
  (IF NOT EXISTS + 버전 UPDATE, 백필 불필요).

### 7.5 핵심 쿼리 예시

```sql
-- Q-HABIT-TODAY — 오늘 확정된 슬롯 집합 (게이지 렌더용)
-- user.db. 앱에서 boolean[4]로 펼침(없는 slot_index = ○). idx_retrieval_day 사용.
SELECT slot_index FROM retrieval_session WHERE local_day = :today;
```

```sql
-- Q-HABIT-STREAK — 연속 달성 일수(4/4 달성한 날만 카운트)
-- "그 날 슬롯 4개 모두 찬 날"을 최신일부터 끊김 없이 센다. 저장 컬럼 없이 로그에서 파생.
WITH full_days AS (
  SELECT local_day
  FROM retrieval_session
  GROUP BY local_day
  HAVING COUNT(DISTINCT slot_index) = 4
)
SELECT local_day FROM full_days ORDER BY local_day DESC;
-- JS: today부터 1씩 내려가며 연속인지 확인해 streak 계산.
--     (오늘이 아직 4/4 전이면 어제부터 연속을 세되, 표시는 "진행 중" 처리)
```

- 스트릭은 저장 컬럼 없이 **로그에서 매번 파생**한다 — 로그가 단일 진실원. 슬롯 로그와 누적값이
  어긋날 여지를 원천 차단하고, 로그만 있으면 언제든 재계산으로 복구 무비용. 규모는 하루 최대 4행이라
  성능 무관.
