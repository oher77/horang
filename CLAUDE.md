# 오케스트레이션 규칙

**적용 범위(중요): 이 섹션은 메인 대화의 오케스트레이터에게만 적용된다. 당신이 서브에이전트(Task/Agent로 스폰된 deep-reasoner/default-worker/task-worker 등)라면 이 섹션 전체를 무시하고 받은 작업을 자신이 직접 수행하라. 서브에이전트가 Agent 도구로 작업을 재위임하는 것은 금지한다 — 재위임 연쇄는 산출물 없이 토큰만 소모한다 (2026-07-07 실제 발생).**

메인 모델(Fable)은 오케스트레이터다. 직접 코드를 수정하지 않는다.
작업을 계획하고, 쪼개고, 서브에이전트에 배분하고, 결과를 종합하는 역할만 한다.

## 작업 배분

- **deep-reasoner** (Opus): 아키텍처 설계, 복잡한 디버깅, 알고리즘 판단 등 무거운 추론
- **default-worker** (Sonnet): 기능 구현, 보일러플레이트, 테스트, 리팩터링 등 일반 코드 작업
- **task-worker** (Haiku): 오타 수정, 파일 정리, 단순 반복 작업 등 가벼운 잡무

## 원칙

- 비싼 모델은 판단에만, 싼 모델은 실행에 쓴다.
- 오케스트레이터의 컨텍스트는 가볍게 유지한다. 긴 파일 읽기와 탐색은 서브에이전트에 위임한다.
- 중요한 판단은 `codex exec`(설치돼 있는 경우)와 deep-reasoner에 병렬로 독립 검토시키고, 서로의 답을 보여주지 않은 채 두 답을 받아 종합한다.
- 예외: 한 줄 수정처럼 위임 오버헤드가 작업 자체보다 큰 경우는 직접 처리해도 된다.
- 앱 품질이 비용 절감보다 우선한다. 품질/신뢰성 확보에 필요한 지출(예: Apple 개발자 계정)은 미루지 말고 필요 시점에 바로 제안할 것.

# 프로젝트: 호랑이 잉글리시 (Horang English)

고등학생 대상 영단어 학습 앱. 구글시트에 정리된 약 2300개 단어(표제어+파생어+동사변형)를 하루 단위 단어장(Day1, Day2...)으로 순차 생성해 암기·복습·테스트하는 흐름.

## 기술 스택 (확정, 2026-07-06)

- **React Native + Expo** (TypeScript)
- DB: `expo-sqlite` (네이티브 SQLite, 관계형 쿼리 필요 — 복습 스케줄, 오답 Top10, 월별 집계 등)
- TTS: `expo-speech` (iOS Safari Web Speech API의 비동기 무음 버그를 피하기 위해 네이티브 필수)
- 애니메이션: `react-native-reanimated` (셀 가리기 파티클 효과)
- 제스처: `react-native-gesture-handler` (기억 단계 좌우 스와이프)
- iOS 빌드: **EAS 클라우드 빌드** (로컬 Xcode 14.2로는 최신 iOS 빌드/제출 불가)
- 개발 실행: **Expo Go** (무료, 실기기 핫리로드)

PWA/네이티브 Swift/Flutter/Capacitor는 검토 후 반려됨 — 근거는 `기술방향.md` §3 참고.

## 개발 환경 제약

- 집 iMac (late 2015, Monterey 12.7.6, Xcode 14.2 한계) — 코드 편집 + Metro 번들러만, 네이티브 컴파일은 EAS 클라우드가 담당
- 학교 Windows 11 PC — RN/Expo 개발 동일하게 가능
- 실기기: iPhone 12 Pro (Expo Go로 QR 연결 테스트)
- **Expo Go에 없는 서드파티 네이티브 모듈 도입 금지가 기본 원칙** — 도입 시 development build(유료 Apple 개발자 계정 필요)가 강제됨. 새 라이브러리 추가 전 Expo Go 호환 여부 확인 필수.

## 데이터 모델

- 단어 원본: `data/voca_data.xlsx` (Sheet1, 2212행, 컬럼: 번호/표제어/파생어/쓰기) → 앱 DB로 import하는 파이프라인 필요 (스키마 미정). 정제 방침은 `기술방향.md` §8 참고 — 번호 컬럼 무시, 완전 빈 행 삭제, **파생어는 표제어와 동등한 독립 단어장 항목**(단어장 노출·시험 출제 모두 동일 취급), 쓰기 컬럼은 영문쓰기(스펠링/동사변형) 문제 데이터로 활용(파싱 규칙은 DB 스키마 설계 시 확정).
- 뜻·예문: 원본 시트에 뜻 컬럼이 없어 품사별 뜻 + 난이도별(고1/2/3, 수능 스타일) 예문을 전 단어 새로 생성해야 함 — **LLM 일괄 사전 생성 → 검수용 중간 산출물 → DB 내장으로 확정** (2026-07-07, 온디맨드/수동 입력 반려)
- 사용자 진행 데이터: 기기 로컬 SQLite, 추후 클라우드 동기화 고도화 예정

## 핵심 도메인 규칙

- 단어장은 하루에 1개만 생성되며, 모든 단어장 간 단어는 중복되지 않는다 (Day1의 단어는 다른 어떤 Day에도 재등장하지 않음)
- 복습 스케줄: 학습일 기준 **-1/-3/-7/-14/-30/-60/-120일** 전 단어장을 최근 순으로 노출
- 기억 인출 실패 아이콘: 5단계(1~5회 실패), 우측 스와이프 시 증가·좌측 스와이프 시 감소
- 낯가림 심한 단어: 오답 횟수 1회 이상인 단어 중 횟수 많은 순 상위 10개
- Income(용돈) 시스템: 실제 결제 연동 없이 점수별 금액을 표시하고 지급 여부만 체크하는 장부 개념 (부모 관리 앱 연동은 추후 기능)

## 화면 구조 (계획)

스플래시 → 단어장(Day 목록, 학습/복습/테스트) / 메뉴(단어장리스트·낯가림 심한 단어·나의 업적·설정)

- **단어장 학습 화면**: 단어/뜻/예문 3컬럼 테이블, 컬럼 헤더 클릭으로 전체 가리기, 셀 클릭 시 파티클(먼지 흩어짐) 효과로 개별 가리기 — 100개 이상 셀 대응을 위해 화면에 보이는 15~20행만 렌더링 + stagger 애니메이션으로 GPU 부하 분산
- **복습 화면**: 뜻/예문이 가려진 상태로 시작
- **테스트 화면**: 문제/답 2컬럼, 채점 시 정답·발음확인·오답·발음헷갈림 컬럼 추가, 점수 확인 시 3초 애니메이션 후 점수 표시
- **나의 업적**: 현재 수준, 이달의 Income, 최근 5일 점수, 낯가림 단어 Top10, 정답 누적 추이, 월별 Income/점수/오답 추이
- **설정**: 난이도(고1/2/3), 하루 학습 단어 개수(변경 시 미진행 단어장만 재생성), 점수별 Income 액수

전체 화면별 상세 요구사항은 `단어장 앱 만들기.md` 참고.
