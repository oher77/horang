## 3. 앱 아키텍처 도식

### 3-1. 2-DB 구조 (설계.md §1)

```
                     iPhone (Expo Go)
      ┌───────────────────────────────────────────┐
      │  app/ 화면들 ──호출──▶ lib/ 쿼리 함수들    │
      │                          │        │       │
      │                 ┌────────▼──┐ ┌───▼─────┐ │
      │                 │content.db │ │ user.db │ │
      │                 │ 읽기 전용 │ │ 읽기+쓰기│ │
      │                 │ 단어·뜻·  │ │ 진행·점수│ │
      │                 │ 예문·쓰기 │ │ ·습관·설정│ │
      │                 └───────────┘ └─────────┘ │
      │                  ▲ 앱에 번들   ▲ 첫 실행 시│
      │                  │ (배포 시    │ 기기에서  │
      │                  │  교체)      │ 생성      │
      └──────────────────┼────────────┼───────────┘
                파이프라인 산출물   마이그레이션은 스키마
                (아래 3-2)          버전 번호로 멱등 실행
```

- 날짜는 전부 정수 epoch day(기기 로컬 자정 기준 일수)로 저장 — 복습 -N일 조회가
  정수 비교 한 번으로 끝난다. 변환은 `lib/dates.ts`의 `toEpochDay()`만 쓴다.

### 3-2. 콘텐츠 데이터 흐름 (설계.md §2–3)

```
data/voca_data.xlsx (원본 🔒)
      │  python3 scripts/clean.py  (몇 번을 돌려도 같은 결과)
      ▼
build/words.json ─── build/id_map.json 🔒 (단어별 영구 ID — 멱등성의 핵심)
      │                └ review/*.csv (사람 검수 큐 → 전량 검수 완료)
      │  scripts/gen_content.py  (Claude Batch API — 뜻·예문 2,430건 생성 완료)
      ▼
build/content.json
      │  scripts/pack_db.py
      ▼
assets/db/content.db  (word 2,416 / meaning 3,719 / example 11,411 / writing 43)
```

절차 재실행이 필요하면 `/content-pipeline` 스킬 참조.

### 3-3. 화면 지도

```
홈 (index)
 ├─▶ 단어장 (day/[dayId]) ◀─┐   학습모드(기본) / 인출모드(카운트다운·사이렌)
 ├─▶ 복습 (review) ─────────┘   복습에서 진입하면 인출모드로 시작
 ├─▶ 테스트 (test)              자기채점 → 점수 → 용돈 장부 기록
 ├─▶ 업적 (achievements)        용돈 장부·차트 5종
 ├─▶ 발음 장부 (pronunciation)
 └─▶ 설정 (settings)
```

## 4. 개발 하네스 시스템 (.claude/ + CLAUDE.md)

Claude Code가 이 프로젝트에서 일하는 방식은 **"비싼 모델은 판단, 싼 모델은 실행"**
원칙으로 설계돼 있고, 모델의 선의에 의존하지 않도록 기계적 안전장치가 이중으로 깔려 있다.

### 4-1. 오케스트레이션 구조

```
사용자 (heruse)
   │  지시
   ▼
╔═══════════════ 메인 대화: Fable (오케스트레이터) ═══════════════╗
║  · CLAUDE.md 규칙 + 개인 메모리(~/.claude/…)를 매 세션 로드     ║
║  · 계획 수립 → 작업 분해 → 워커 스폰 → 결과 검토·종합          ║
║  · 직접 코드 수정은 "한 줄급" 예외만                            ║
║                                                                 ║
║     스폰 시 7요소 계약 프롬프트 (재위임 금지, 필독 섹션,        ║
║     산출물 명세, 파일 소유권, 검증 명령, 편차 보고, 중단 규칙)  ║
║        │                                                        ║
║        ├─▶ deep-reasoner   (Opus, effort xhigh)                 ║
║        │     아키텍처 설계·난해한 디버깅·알고리즘 판단          ║
║        ├─▶ default-worker  (Sonnet, effort high)  ← 주력        ║
║        │     기능 구현·리팩터링·테스트 작성                     ║
║        │     · horang-worker-guide 스킬 자동 주입(공통 규약)    ║
║        │     · .claude/agent-memory/ 에 경험 축적               ║
║        └─▶ task-worker     (Haiku)                              ║
║              오타·이름변경·단순 반복 잡무                       ║
║                                                                 ║
║  교차검증: 중요 판단은 codex CLI(~/.npm-global/bin/codex)와     ║
║  deep-reasoner에 서로 답을 안 보여주고 병렬 질의 → 종합         ║
║  검증자 분리: 중요 구현은 구현 워커와 다른 신선한 에이전트가    ║
║  검증 항목 통과 여부만 재확인                                   ║
╚═════════════════════════════════════════════════════════════════╝
```

에이전트별 모델·effort는 `.claude/agents/*.md`의 frontmatter에 고정돼 있다
(스폰 프롬프트로는 못 바꿈 — 조정하려면 파일 수정).

### 4-2. 기계적 안전장치 (settings.json)

모델이 깜빡하거나 잘못 판단해도 시스템이 잡아주는 층.

```
Claude가 Edit/Write 시도
   │
   ├── permissions.deny 검사 ──✗ 차단──▶  data/** (원본 엑셀)
   │                                      build/id_map.json (영구 ID 맵)
   │                                      단어장 앱 만들기.md (기획서 원문)
   │        스크립트(clean.py 등)를 경유한 쓰기는 허용 — 사람/모델의 직접 수정만 차단
   │
   └── 턴 종료 시 (Stop/SubagentStop 훅)
          └─▶ .claude/hooks/tsc-check.sh 가 수정된 .ts/.tsx가 있으면 npx tsc --noEmit 1회 실행
              실패하면 모델에게 자동 통보 → 워커가 검증을 깜빡해도 잡힌다
              (편집마다 검사하던 방식은 다중 편집의 중간 상태 오탐이 잦아 2026-07-11 턴 종료 검사로 변경)
```

### 4-3. 스킬 (필요할 때만 로드되는 절차서)

| 스킬 | 용도 |
|---|---|
| `horang-worker-guide` | 코드 작업 공통 규약 (스택 고정, Expo Go 가드레일, 검증 명령, 보고 형식). default-worker 스폰 시 자동 주입 |
| `content-pipeline` | 뜻·예문 배치 생성 절차 (제출→확인→회수→검증→DB 빌드) |

### 4-4. 설정 파일 각각의 역할

| 파일 | 역할 | git 추적 |
|---|---|---|
| `CLAUDE.md` | 작업 규칙·프로젝트 현황 (매 세션 로드) | O |
| `.claude/settings.json` | 권한 deny + 훅 (팀 공통) | O |
| `.claude/settings.local.json` | 개인 환경변수 (API 키 등) | X |
| `.claude/launch.json` | 프리뷰 서버 실행 정의 | O |
| `.claude/agents/*.md` | 에이전트 3종의 모델·effort·역할 정의 | O |
| `~/.claude/projects/…/memory/` | Fable의 세션 간 개인 메모리 (저장소 밖) | — |

### 4-5. 이렇게 굴러가는 이유 (요약)

- **비용**: 토큰 비용에 민감 → Fable/Opus는 판단에만, Sonnet/Haiku가 실행.
- **품질**: 싼 모델의 품질 하한은 프롬프트 구조가 결정 → 7요소 계약 + 편차 보고 강제.
- **안전**: 원본 데이터와 영구 ID는 모델이 실수해도 못 건드리게 시스템 층에서 차단.
- **재발 방지**: 사고가 나면 CLAUDE.md의 실패 사례 체크리스트에 추가해 규칙화.

## 5. 자주 쓰는 명령

```bash
npx expo start             # Metro 개발 서버 (iPhone Expo Go에서 QR로 연결)
npx tsc --noEmit           # 타입체크 (훅이 자동으로도 돌림)
python3 scripts/clean.py   # 원본 정제 파이프라인 (멱등, 인자 없음)
```

- Expo SDK **54 고정** (App Store의 Expo Go 호환 한계 — 업그레이드 금지)
- iOS 네이티브 빌드는 로컬 불가 → EAS 클라우드 빌드. 개발 반복은 Expo Go로 무료.
