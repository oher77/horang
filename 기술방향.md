# 호랑이 잉글리시 — 기술 방향 결정서

작성일: 2026-07-06 · 상태: **스택 확정** (사용자 반론 + deep-reasoner 독립 검토 일치)

## 1. 결정 요약

**React Native + Expo** (TypeScript)

| 영역 | 선택 | 비고 |
|---|---|---|
| DB | expo-sqlite | 네이티브 SQLite, 풀 SQL, 축출 리스크 없음 |
| TTS | expo-speech | AVSpeechSynthesizer 직접 사용, 제스처 제약 없음 |
| 애니메이션 | react-native-reanimated | 셀 가리기 파티클 효과, Expo Go 포함 |
| 제스처 | react-native-gesture-handler | 기억 단계 좌우 스와이프, Expo Go 포함 |
| iOS 빌드 | EAS 클라우드 빌드 | 낡은 Xcode(14.2) 우회 |
| 개발 실행 | Expo Go | 무료, 아이폰 실기기 핫리로드 |

## 2. 결정 근거 (핵심 3논점)

1. **TTS 신뢰성** — iOS Safari Web Speech API는 첫 재생을 사용자 탭 핸들러 안에서 동기 호출해야 하고, 비동기 이후 호출 시 무음이 되는 버그가 흔함. "단어 탭 → 발음"이 매일 반복되는 핵심 동작이고, 시험 화면의 발음 확인은 채점 흐름과 얽혀 비동기를 타기 쉬워 웹으로는 구조적으로 취약. expo-speech는 이 제약 자체가 없음.
2. **관계형 쿼리** — 오답 상위 10개(`ORDER BY ... LIMIT 10`), 월별 Income 합계(`GROUP BY SUM`), -1/-3/-7/-14/-30/-60/-120일 복습 스케줄(날짜 오프셋 조회)은 전형적인 SQL 작업. IndexedDB는 부적합, wa-sqlite+OPFS는 iOS 신뢰성이 낮음.
3. **데이터 영속성 + 배포 경로** — iOS는 PWA 저장소를 공격적으로 축출함(몇 주 미사용 시 학습 이력 통삭제 가능). 앱스토어 정식 배포 경로도 PWA에는 없음. 네이티브 앱 샌드박스 SQLite는 두 문제 모두 없음.

## 3. 반려된 대안

- **PWA**: 위 3논점 모두에서 충돌. 무료·즉시 배포 장점은 Expo Go가 상쇄.
- **네이티브 Swift**: 집 iMac(late 2015, Monterey 12.7.6)은 Xcode 14.2가 한계 → 최신 iOS 디버깅·앱스토어 제출(Xcode 15+ 요구) 불가. 학교 Windows PC 병행도 불가.
- **Flutter**: 가능하나 "맥 없는 iOS 빌드 + Expo Go 즉시 실행" 수준의 무료 경험이 없고 학습 곡선만 추가.
- **Capacitor**: 결국 WebView라 TTS/저장소의 웹 리스크를 그대로 상속.

## 4. 개발 워크플로 / 환경

- **집 iMac**: 코드 편집 + Metro 번들러만 담당 (이 사양으로 충분). 네이티브 컴파일은 EAS 클라우드가 수행 → Xcode 14.2 한계 무력화.
- **학교 Windows 11**: RN/Expo 개발 완전 동일하게 가능. iOS 빌드는 EAS 클라우드라 OS 무관.
- **iPhone 12 Pro**: Expo Go 앱 설치 → QR 연결 → 실기기 핫리로드 테스트.

## 5. 비용 전략

- 개발 전 기간: **Expo Go만으로 0원**. 핵심 의존성(expo-sqlite, expo-speech, reanimated, gesture-handler) 전부 Expo Go 포함.
- **Apple 개발자 계정($99/년)이 필요해지는 시점**: ① 독립 앱으로 7일 초과 장기 설치, ② TestFlight/앱스토어 제출. 그 전까지 지불하지 않는다.
- EAS 무료 티어 빌드 한도: 개발 반복은 Expo Go로 처리해 클라우드 빌드를 드물게 유지하면 무료 한도로 충분.

## 6. 기술 가드레일

- **Expo Go에 포함되지 않은 서드파티 네이티브 모듈 도입 금지가 기본 원칙.** 도입하는 순간 development build(= $99 계정)가 강제됨. 새 라이브러리는 도입 전 Expo Go 호환 여부를 반드시 확인하고, 비호환이면 도입 전에 논의.
- TTS 연속 탭 시 중복 재생 방지: `Speech.stop()` 후 `speak()` 패턴.
- 날짜 컬럼은 정렬·범위 인덱스가 걸리도록 저장 포맷 통일 (복습 스케줄 조회 대비).

## 7. 데이터 관련 확정 사항

- **예문의 로컬 SQLite 저장: 무리 없음.** 2300단어 × 3난이도(고1/2/3) × 품사별 예문이라도 수 MB 수준. (기획서의 검토 요청 항목)
- 단어 원본: 구글시트(약 2300단어) → 앱 DB로 import하는 파이프라인 필요.
- 사용자 진행 데이터: 기기 로컬 SQLite, 추후 클라우드 동기화 고도화 (기획서 방침 유지).

## 8. 구글시트 원본 데이터 정제 방침 (2026-07-07 확정)

`data/voca_data.xlsx` (Sheet1, 2212행, 컬럼: 번호/표제어/파생어/쓰기)를 확인한 결과 및 처리 방침:

- **번호 컬럼**: 162행까지만 채워져 있고 이후 비어있음(순번 매기기 중단). Day 구성은 랜덤 추출이라 **무시**.
- **완전 빈 행** (4개 컬럼 전부 NaN, 8곳): **삭제**.
- **파생어 컬럼 (205개, 205행 전부)**: 표제어와 짝을 이루는 일반 케이스(`abort → (abortion)`)든 표제어 없이 파생어만 있는 케이스(`(embody)`, `half­moon`)든 관계없이, **파생어는 표제어와 동일한 자격의 독립 단어장 항목**이다. 즉 단어장 노출·시험 출제 모두에서 표제어와 동등하게 취급한다(기획서 원문: "표제어와 파생어가 단어장에 나와야할 단어"). 표제어-파생어 쌍 관계(파생 출처) 자체는 참고 메타데이터로만 남기고, 노출/출제 로직에서 우열을 두지 않는다.
- **쓰기 컬럼** (43개, 형식 혼재 — `두배의: d______`처럼 스펠링 힌트, `eat의 과거형`/`catch의 동사변형`처럼 문법 메모, `가위`처럼 뜻만 있는 경우 등): **영문쓰기(스펠링/동사변형) 문제 데이터로 활용**. 구체적 파싱 규칙(형식별 목표 단어 추출 방식 등)은 §9 DB 스키마 설계 시 함께 정한다.

## 9. 콘텐츠 준비 방식 (2026-07-07 확정)

- **원본 시트에 '뜻' 컬럼이 없음** (번호/표제어/파생어/쓰기 4컬럼뿐). 따라서 품사별 뜻 + 난이도별(고1/2/3) 예문을 2200여 단어 전부 새로 준비해야 함.
- **확정: LLM 일괄 사전 생성 → DB 내장.** 개발 단계에서 배치 생성하고, 검수 가능한 중간 산출물(CSV/JSON)을 거쳐 DB에 넣는다. 앱은 오프라인 완결, 런타임 API 비용 0. 생성 방법(Claude 세션 내 배치 vs API 스크립트)은 설계 단계에서 확정.
- 반려: 앱 내 온디맨드 생성(네트워크 의존·API 키 관리·런타임 비용·지연), 수동 입력(분량 비현실적).
